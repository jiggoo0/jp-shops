## 🛠 มาตรฐานการตรวจสอบคุณภาพโค้ด (Quality Control)

เพื่อให้โค้ดเบสมีความสะอาดและเป็นระเบียบตามมาตรฐาน Senior Engineer ให้ปฏิบัติตามขั้นตอนดังนี้:

### 1. การตรวจสอบรูปแบบโค้ด (ESLint)

- **คำสั่ง**: `pnpm run lint`
- **แนวทาง**: หากพบข้อผิดพลาด (error) หรือคำเตือน (warning) ต้องวิเคราะห์สาเหตุและแก้ไขที่ต้นตอตามหลัก Best Practice **ห้ามใช้การปิด (disable) กฎของ linter เด็ดขาด**

### 2. การจัดรูปแบบโค้ด (Prettier)

- **คำสั่ง**: `pnpm run format`
- **แนวทาง**: รันหลังจากแก้ไขปัญหาจาก ESLint เรียบร้อยแล้ว เพื่อให้ฟอร์แมตของโค้ดสอดคล้องกันทั้งโปรเจกต์

### 3. การค้นหาโค้ดที่ไม่ได้ใช้งาน (Knip)

- **คำสั่ง**: `pnpm run unused` (หรือ `pnpm knip`)
- **แนวทาง**: ตรวจสอบหาไฟล์หรือตัวแปรที่ไม่ได้ใช้งาน หากพบให้วิเคราะห์ความเกี่ยวข้องและลบออกอย่างปลอดภัยเพื่อลดภาระของระบบ

### 4. การตรวจสอบความถูกต้องของ Type (TypeScript Check)

- **สำคัญมาก**: ลินท์ (Lint) ตรวจสอบแค่รูปแบบ แต่ไม่ได้ตรวจสอบว่า Data Type ผิดพลาดหรือไม่
- **คำสั่ง**: `pnpm run type-check` (ปกติคือ `tsc --noEmit`)
- **แนวทาง**: ต้องรันก่อนที่จะ Build เสมอ เพื่อป้องกัน Runtime Error ที่เกิดจากการส่งค่าผิดประเภท

### 5. การทดสอบซอฟต์แวร์ (Testing)

- **แนวคิด**: โค้ดที่สะอาดแต่ทำงานผิดพลาดก็ไม่มีความหมาย
- **คำสั่ง**: `pnpm run test` (Unit Test) และ `pnpm run test:e2e` (End-to-End)
- **แนวทาง**: เน้นการทำ Regression Testing เพื่อให้แน่ใจว่าการแก้ไขโค้ดใหม่ไม่ไปกระทบฟีเจอร์เดิมที่ทำงานได้อยู่แล้ว

### 6. การตรวจสอบช่องโหว่ความปลอดภัย (Security Audit)

- **คำสั่ง**: `pnpm audit`
- **แนวทาง**: ตรวจสอบ Library ภายนอกว่ามีช่องโหว่ (Vulnerabilities) หรือมีปัญหาด้านลิขสิทธิ์หรือไม่ หากพบต้องอัปเดตเวอร์ชันตามคำแนะนำทันที

### 7. การตรวจสอบประสิทธิภาพการ Build (Build Verification)

- **คำสั่ง**: `pnpm run build`
- **แนวทาง**: ก่อนจะส่งงาน (Push) ควรลอง Build จริงในเครื่องตัวเองก่อนหนึ่งรอบ เพื่อให้แน่ใจว่าไม่มี Error แปลกๆ ที่เกิดขึ้นเฉพาะตอน Compile เป็น Production
